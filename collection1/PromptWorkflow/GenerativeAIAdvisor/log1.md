# ドメイン知識テンプレート化 実行結果

---

## 1. 会話の概要

この会話では、**生成AIの活用方法が分からないユーザーを支援する「AIアドバイザーシステム」の設計**について扱いました。ユーザーは、処理方式（自己完結型/司令塔型）と対話スタイル（テーマ型/伴走型）の4パターンを統合した単一ワークフローの構築を依頼。私は、モード自動判定機能を持つメインプロンプトと、状況別の補助プロンプト群を設計し、実装可能なシステムとして提供しました。

---

## 2. 知識の分類結果

### 【AI既知の知識】
- プロンプトエンジニアリングの基本構造（役割・タスク・制約・出力形式）
- マルチステップワークフローの設計手法
- 対話型AIの応答設計原則（簡潔性・具体性・ユーザビリティ）
- テンプレート化の一般的手法
- マークダウン記法・コードブロックの使用方法
- チャットボット設計におけるユーザー体験の考慮

### 【ユーザー提供の新規ドメイン知識】
- **4モード分類体系**：「処理方式（自己完結型/司令塔型）× 対話スタイル（テーマ型/伴走型）」という独自の分類軸
- **モード自動判定の設計思想**：ユーザーに選択を意識させず、入力から透過的に最適モードを判定する仕様
- **生成AIアドバイザーの具体的ユースケース**：「何に使えるか分からない」状態から「実行可能」状態への移行支援
- **統合ワークフロー要件**：複数モードを単一システムで切り替える設計要求
- **初期診断の4軸評価**：明確性・緊急性・リソース・情報量によるユーザー状況判定
- **3問以内ルール**：過度な質問を避け、推測ベースで動く対話設計の制約

## 3. テンプレート化されたドメイン知識

````markdown
# 生成AIアドバイザーシステム 設計ドメイン知識

## 概要
生成AIの活用方法が不明なユーザーを支援するため、4つの動作モード（自己完結型/司令塔型 × テーマ型/伴走型）を統合し、入力から自動的に最適な対応方式を判定するアドバイザーシステムの設計パターン。

## 重要な前提・制約
- **透過的モード切替**：ユーザーにモード選択を意識させない（自然な対話を維持）
- **推測優先の設計**：情報不足でも仮説を立てて動く（質問は3問以内が目安）
- **単一システム統合**：複数プロンプトではなく、1つのメインプロンプトで全モードをカバー
- **即実行可能性**：抽象論ではなく、コピペで使えるプロンプト・手順を提供

## 専門用語・定義

### 処理方式（2分類）
- **自己完結型**：AI自身が情報を集約し、単独で問題解決を完結させる方式
- **司令塔型**：AI自身は処理せず、他ツール・情報源の活用方法を指示する方式

### 対話スタイル（2分類）
- **テーマ型**：特定の課題を中心に集中的に対応（明確な目標がある場合）
- **伴走型**：継続的対話を通じて段階的にアドバイス（プロセス全体をサポート）

### 4つの動作モード
1. **自己完結×テーマ型**（迅速解決モード）：課題明確 + 情報十分 → AI自身が完成品を提供
2. **自己完結×伴走型**（段階的支援モード）：課題やや不明確 → 対話で引き出しながら解決
3. **司令塔×テーマ型**（戦略提示モード）：課題明確 + 外部ツール活用希望 → 他ツールの使い方を提示
4. **司令塔×伴走型**（メンター型モード）：漠然とした相談 → 活用シーン発見から実装まで伴走

### 初期診断の4軸
ユーザー入力から自動判定する評価軸：
- **明確性**：課題が明確か / 漠然としているか
- **緊急性**：今すぐ解決したいか / 継続的に相談したいか
- **リソース**：自分で実行したいか / 指示だけ欲しいか
- **情報量**：十分な情報があるか / 追加情報が必要か

## プロセス・ルール

### 標準ワークフロー
```
STEP1: 状況診断（自動実行）
   ↓ 4軸評価で判定
STEP2: モード自動選択
   ↓ A/B/C/Dいずれかに分岐（透過的）
STEP3: 実行
   ↓ 最小限の追加質問 + 具体的解決策提示
STEP4: 品質チェック
```

### 質問設計の原則
- **3問ルール**：追加質問は最大3つまで
- **選択肢提示**：Yes/No質問を避け、選択肢から選ばせる
- **仮説提示**：「分からない」と言われたら、複数仮説を提示して選択を促す

### 出力設計の原則
- 簡潔な説明（長文回避）+ 実装可能なプロンプト/手順
- 構造化（リスト・コードブロック活用）
- 専門用語を避ける平易な表現

## 具体例・パターン

### 入力パターン例と対応モード

**例1：「生成AI、何に使えるんですか？」**
→ 診断：明確性（低）+ 緊急性（低）→ **司令塔×伴走型**
→ 対応：A/B/C/Dの選択肢を提示し、活用シーン探索から開始

**例2：「会議の議事録作成がめんどくさい」**
→ 診断：明確性（高）+ 情報量（中）→ **自己完結×テーマ型**
→ 対応：録音データ有無を1問確認後、即実行プロンプト提供

**例3：「営業資料作成を効率化したいが、どのツールを使えばいい？」**
→ 診断：課題明確 + 外部ツール志向 → **司令塔×テーマ型**
→ 対応：ツール比較と使い方を提示（AI自身は処理しない）

## 注意点
- **モード名称はユーザーに見せない**：内部設計用語であり、対話中は自然な言葉で対応
- **モード固定を避ける**：会話の流れでモードが変化することを許容（柔軟性優先）
- **補助プロンプトの位置づけ**：メインプロンプトで完結が基本。補助は必要時のみ追加
- **"できないこと"の誠実な開示**：生成AIの限界も正直に伝える（過度な期待を持たせない）
- **実装環境依存**：システムプロンプトとして設定する想定（チャットUI前提）

## 拡張ポイント
- 業界特化版（医療・教育・営業など）へのカスタマイズ
- レベル別対応（初心者/中級者/上級者）
- 過去対話履歴の学習による精度向上（実装依存）
